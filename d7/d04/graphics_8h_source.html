24 /** Encapsulates the superset of drawing attributes for all supported primitives and draw calls. These include
25 the primary fill color, the use of gradient fill or not, the secondary fill color and fill direction used
26 by the gradient, a texture image to be blended with the underlying color, the outline color and width and
27 the opacity for the fill and outline colors. The use of these attributes is demonstrated in the example
28 image below:
29
30 \image html brush1.jpg
31 */
32 struct Brush
33 {
34 float fill_color[3] = {1.f,1.f,1.f}; ///< The primary fill color for shapes.
35 ///<
36
37 float fill_secondary_color[3] = { 1.f,1.f,1.f }; ///< The fill color used as a secondary one when using a color gradient.
38 ///<
39
40 float fill_opacity = 1.0f; ///< The opacity (1-transparency) of the primary fill color. A (maximum)
41 ///< value of 1.0f means "fully opaque", while a value of 0.0f means
42 ///< fully transparent (invisible).
43
44 float fill_secondary_opacity = 1.f; ///< The opacity of the second fill color, used for the gradient.
45 ///<
46
47 float outline_color[3] = { 1.f,1.f,1.f }; ///< The color of the shape outline (where applicable). Most drawn
48 ///< shapes, like disks, line segments and rectangles use this
49 ///< property to set the outline stroke color.
50
51 float outline_opacity = 1.f; ///< The opacity of the outline. Maximum value is 1.0f (fully opaque),
52 ///< and the minimum value is 0.0f (fully transparent).
53
54 float outline_width = 1.0f; ///< The stroke width in pixel units. Can accept fractional values.
55 ///<
56
57 std::string texture = ""; ///< The filename of a bitmap to project over and blend with the
58 ///< underlying color of the fill. The image is multiplicatively
59 ///< combined with the entire fill motif, respecting also the opacity.
60 ///< Currently, only PNG images are supported, with transparency.
61 ///< Images of arbitrary size are supported but are internally upscaled
62 ///< to the nearest power of two in each dimension, using linear
63 ///< interpolation. The bitmap, regardless of its aspect ratio,
64 ///< covers the entire shape end to end, according to its own
65 ///< parameterization. This means that a square image drawn on a non-square
66 ///< rectangle will stretch the image. To avoid this, the drawn rectangle
67 ///< should follow the aspect ratio of the image. By default, no image is
68 ///< used. When an image filename is provided, it is loaded once and
69 ///< internally cached for repeated use.
70 ///<
71 ///< Keep in mind that when using a bitmap on a drawable shape, the
72 ///< image parametric space (expressed as u,v coordinates in the image below)
73 ///< is mapped to corresponding parameters of the underlying shape.
74 ///< For example, for the disk shape, the horizontal axis of the image is
75 ///< mapped to the circumferance of the disk, while the height is mapped to
76 ///< the disk radius.
77 ///< \image html uv.jpg
78
79 bool gradient = false; ///< Enables or disables the gradient fill of a shape.
80 ///<
81
82 float gradient_dir_u = 0.0f; ///< The gradient by default uses the v dimension of the parametric shape.
83 ///< In practice though, the calculation is as follows: s = u*gradient_dir_u + v*gradient_dir_v,
84 ///< where s is the color mixing parameter, clamped to [0.0,1.0].
85 ///< This means that we can change the linear parameters gradient_dir_u/v to bias the
86 ///< between the two parametric space axes.
87 ///< For example:
88 ///< \image html gradient.jpg
89
90 float gradient_dir_v = 1.0f; ///< \see gradient_dir_u
91 };

695 /** Loads all PNG bitmaps located in a directory specified by the argument dir at once.
696
697 Preloading image assets prior to actually requesting a draw call with a brush that
698 uses them can significantly boost performance during the typical draw call when a
699 bitmap is first used.
700
701 When a named image is used for a brush, the first time the
702 library encounters the image asset with the specific unique name, it attempts to
703 load the image, scale the bitmap to the closest power of two (if
704 not already in the right resolution) and create the necessary internal representation
705 for the underlying graphics API. For large assets, this process can take some time,
706 noticeable as momentary freeze and general slowdown, depending on the size and number of
707 images needed to be loaded. This of course happens only the first time a bitmap is
708 used in the application, but can nevertheless degrade the user experience.
709
710 preloadBitmaps performs the above process in bulk, for all PNG bitmaps found in the
711 provided directory. This can be done during the application initialization stage,
712 before the actual main interaction loop begins. The function can be called multiple
713 times for a different directory each time, to progressively load organized collections
714 of bitmaps. preloadBitmaps returns a vector of the bitmaps successfully loaded, which
715 can be used as a means to validate that all assets have been properly loaded. Another useful
716 application of the function is to load multiple bitmaps without the need to know the
717 specific names beforehand. For example, one can organize the frames of an animation sequence
718 into a separate folder and load all the sequence at once, maintaining the animation bitmap
719 names for reference and cycling over the animation sequence. Another different application
720 example is the call of the function on a directory to discover and load all images contained
721 there to be able to list and preview the images during runtime (e.g. in an image viewer
722 application).

725 after the call to createWindow.
